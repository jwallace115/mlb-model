### When They Run

All of these run in **every pipeline execution** — 7AM, 11AM, 12PM, 5PM:

- `build_pitcher_db()` is called at the start of `run()` in run_model.py AND in `refresh_games()` in refresh.py
- `build_offense_db()` is called at the start of both
- Both use **per-day caching**: results are cached to `data/cache/pitchers_v2_YYYY-MM-DD.json` and `data/cache/offense_v2_YYYY-MM-DD.json`
- The cache is checked FIRST — if today's cache exists, the API is not called again
- This means: **the first run of the day sets the cache, and all subsequent runs use that cache**

### Root Cause Identified

**The offense cache poisoning:**

1. Manual run at 4:05 AM called `build_offense_db()`
2. FanGraphs returned 403 for both 2026 and 2025
3. Savant fallback for 2026 returned only 2 teams (NYY, SFG — the only teams who'd played by that point)
4. 2 teams is a non-empty dict → **cached immediately**
5. The function never fell through to try Savant 2025 (which has all 30 teams)
6. All subsequent runs today will read the 2-team cache

**Yesterday at 7:00 AM**, Savant 2026 had zero teams (pre-Opening Day), so the function fell through to Savant 2025, which returned all 30 teams. That's why yesterday had good data.

**By this morning (4:05 AM)**, Savant 2026 had 2 teams from the single Opening Night game. That partial result poisoned the cache.

**Later check (just now):** Savant 2026 now has **24 teams** from yesterday's 11-game slate. But the values are extremely noisy (1-game samples: MIL=140.2, ARI=72.0). These would be worse than the stable 2025 values.

## Task 5 — Would a Manual Refresh Fix This?

**NO — not by itself.** Running the pipeline again would read today's cache (the 2-team version) and NOT re-fetch.

**To fix, you would need to:**
1. **Delete** `data/cache/offense_v2_2026-03-27.json`
2. Then re-run the pipeline
3. FanGraphs would still 403
4. Savant 2026 would return 24 teams with noisy 1-game values (bad)
5. OR: modify the fallback logic to prefer 2025 when 2026 has < N teams

**The correct fix is:** modify `build_offense_db()` to require a minimum team count (e.g., 25+ teams) before caching, and fall through to prior year if the current year has too few teams. This prevents early-season cache poisoning.

**For a quick workaround right now:** delete today's offense cache and manually place yesterday's 30-team cache file as today's:
```bash
cp data/cache/offense_v2_2026-03-26.json data/cache/offense_v2_2026-03-27.json
```
This would give all 30 teams the 2025 season values — much better than 2 teams.
